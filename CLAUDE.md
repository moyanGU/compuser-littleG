# compuser 工作区规则

本文件保留为 Claude/Codex 协作时的工作规则说明，不作为产品边界、支持声明或发布依据的权威来源。相关事实应分别以 `ARCHITECTURE.md`、`PHASE5_VERIFIED_SUPPORT_ENVELOPE.md` 和 `apps/web-panel/WEB_PANEL_BOUNDARIES.md` 为准。

本文件是 `compuser` 项目全过程必须遵守的工作规则。它优先于个人偏好，指导所有实现、重构、调试、测试和文档编写行为。

## 1. 核心原则

### 1.1 编码前先思考

- 不要假设用户意图，不要用主观猜测补全需求。
- 遇到歧义时，先列出假设、选项和权衡，再开始实现。
- 发现更简单的路径时，应明确指出，而不是默认走更复杂方案。
- 如果关键信息不足，先停下来澄清，不允许带着困惑继续实现。

### 1.2 简单优先

- 只实现当前任务需要的最小可行方案。
- 不为未来可能发生的需求预埋复杂抽象。
- 不引入未经要求的可配置性、插件化或泛化层。
- 如果一个设计让资深工程师觉得“明显过度设计”，就继续简化。

### 1.3 外科手术式修改

- 只修改与当前请求直接相关的文件和代码。
- 不顺手重构、不顺手改注释、不顺手改风格、不顺手清理历史问题。
- 只清理由本次改动直接引入的遗留项，例如未使用 import、变量或函数。
- 对既有代码保持仓库原生风格，优先服从当前仓库已有约定。

### 1.4 目标驱动执行

- 所有非琐碎任务必须先定义成功标准，再开始实现。
- 多步骤任务必须先写简短计划，格式为 `步骤 -> 验证`。
- Bug 修复必须优先复现，再修复，再验证回归。
- 实现完成后必须用测试、脚本、命令输出或可观察行为验证结果。

## 2. compuser 项目特定规则

- `compuser` 是一个 Windows 桌面系统 agent，不是单纯的脚本集合。
- 架构目标是：
  - 以 Query Engine 为核心循环。
  - 以 Harness 负责上下文、记忆、安全、预算和压缩。
  - 以工具系统承载桌面控制、文件操作、搜索、命令执行和能力发现。
  - 以 Windows-MCP 作为桌面执行底座。
  - 以 CLI-Anything 的方法论把高价值能力封装为 agent-native 接口。
- 优先真实后端，不优先 GUI 自动化：
  - 能通过 CLI、API、SDK、项目文件直接完成的动作，不先走截图点击。
  - GUI 自动化是兜底路径，不是默认主路径。
- 先观测再操作：
  - 跨应用动作前，优先获取桌面状态、窗口状态、UI 结构或截图。
- 工具输出必须受预算约束：
  - 大输出优先生成摘要和指针，不直接灌满上下文。
- 记忆只存结构化事实：
  - 项目结构、用户偏好、任务进度、关键发现应进入记忆目录。
  - 不把记忆系统退化为冗长的散文摘要。

## 3. 安全规则

- 生成和执行必须分离：
  - 主 Agent 负责提出动作。
  - 权限和安全层负责决定是否允许执行。
- 以下操作默认视为高风险：
  - 删除文件或目录。
  - 执行会修改系统状态的 PowerShell 命令。
  - 安装、卸载或启动系统级软件与服务。
  - 写注册表、改环境变量、批量输入、跨应用自动化。
- 高风险动作必须具备明确理由、作用范围和可验证目标。
- 对系统进行桌面自动化时，应尽量先聚焦目标窗口，避免误操作后台应用。

## 4. 参考源码使用原则

- `claude_code_src-master` 主要用于学习和借鉴架构边界，不做无目的重构。
- `Windows-MCP-main` 主要用于借鉴 Windows UI 自动化、快照、窗口控制与输入执行能力。
- `CLI-Anything-main` 主要用于借鉴 harness 方法论、CLI/Skill 组织方式、测试与发布规范。
- 借鉴时优先抽取设计与接口，不盲目照搬全部实现细节。

## 5. 实施要求

- 开始实现前，必须回答四个问题：
  - 目标是什么？
  - 歧义在哪里？
  - 最小改动是什么？
  - 如何验证完成？
- 新增模块时，优先明确以下边界：
  - 职责
  - 输入输出
  - 依赖关系
  - 失败模式
  - 权限级别
- 新增工具时，至少定义：
  - 名称与用途
  - 输入 schema
  - 输出格式
  - 权限策略
  - 并发安全属性
  - 搜索提示词

## 6. 文档与测试

- 对核心架构改动，必须同步更新 `ARCHITECTURE.md`。
- 对高风险能力，优先补最小且有价值的测试。
- 不写低价值的样板测试，不用测试重复实现细节。
- 文档必须描述真实实现，不允许写与当前代码不一致的说明。
