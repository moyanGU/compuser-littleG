# Compuser 真实可用交付说明（2026-05-31）

## 1. 本轮交付结论

本轮交付目标已完成：

- `web-panel` 与 `npm run dev` 默认走真实 provider，不再隐式回落到 `demo`
- provider 缺配置、不可达、结构化返回错误时，失败会显式暴露
- shell smoke 与真实 provider smoke 已分离，验证口径不再混淆

本轮交付只收敛“当前支持边界内的真实可用”，不扩大支持矩阵，不刷新 Phase 5 已发布 claim。

## 2. 当前正确配置

如果使用云雾接口，当前项目里的配置语义应为：

```powershell
$env:COMPUSER_MODEL_PROVIDER='openai-compatible'
$env:COMPUSER_MODEL_NAME='gpt-5.4-mini'
$env:COMPUSER_MODEL_BASE_URL='https://yunwu.ai/v1/chat/completions'
$env:COMPUSER_MODEL_API_KEY='***'
```

注意：

- `COMPUSER_MODEL_PROVIDER` 不是模型名；本轮仍只支持 `demo` 与 `openai-compatible`
- `https://yunwu.ai/` 站点根地址不是当前客户端需要的 API 地址；必须使用完整 `chat/completions` 路径

## 3. 本轮实际落地项

### 3.1 入口收敛

- `npm run dev` 默认 provider 已改为 `openai-compatible`
- `web-panel` 默认 provider 与 CLI 走同一套 provider 解析和 model options 构造路径
- `demo` 仅保留为显式测试/演示模式

### 3.2 失败可见性

- 缺配置：显式暴露 `missing_dependency`
- endpoint 不可达：显式暴露 `transport_error`
- provider 结构化错误：显式暴露 `provider_error`

### 3.3 网关兼容性修复

- 已修复部分 OpenAI-compatible 网关拒绝 `.` 工具名的问题
- 内部工具名保持不变
- 对外请求自动做可逆工具名编码/解码

## 5. 发布口径

本轮可以对外使用的准确口径是：

- 当前 `compuser` 的 `web-panel` 与 `npm run dev` 已默认走真实 provider
- 当前支持边界内，真实 provider 失败不会被静默降级掩盖
- `web:panel:shell-smoke` 不代表真实 provider 已验证
- 真实 provider 验证以 `web:panel:provider-smoke` 或 `product:verify` 为准

本轮不应对外宣称的内容：

- 不应宣称支持边界已扩大
- 不应宣称已刷新 Phase 5 已发布 scorecard
- 不应把本轮入口整改表述成新的 live support claim

## 6. 发布前最后检查

如果只是交付本轮整改结果，建议按下面顺序复核：

1. `npm run dev:regression`
2. `npm test`
3. `npm run web:panel:provider-smoke`
4. `npm run product:verify`

如果要刷新发布证据而不是仅交付整改说明，额外执行：

5. `npm run phase5:scorecard -- --runs 1 --cooldown-ms 1000 --max-total-minutes 5 --endpoint http://127.0.0.1:8010/mcp --permission-mode auto --json-out E:\compuser\compuser\artifacts\scorecard\phase5-latest.json`

执行第 5 步前必须再次确认：

- 当前 endpoint 与已发布 artifact 要么一致，要么你准备同步刷新发布口径
- 本轮目标仍然只是“入口真实可用整改”，不是无边界扩大发布

## 7. 手工验收注意事项

- 中文任务的手工 HTTP 提交，优先用面板 UI 或 Node/UTF-8 客户端
- 不要用未经确认编码行为的 PowerShell `Invoke-WebRequest` 结果直接判断面板中文链路是否损坏
- 本轮已确认：之前出现的中文乱码判断来自手工提交端编码，不是 `web-panel` 服务端链路问题
