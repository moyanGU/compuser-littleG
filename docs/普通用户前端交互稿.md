# 普通用户前端交互稿

本文档定义 `compuser` 普通用户入口的前端交互方案。

它只描述三件事：

- 首屏应该如何组织
- 用户提交任务后应进入哪些状态
- 每种状态下用户应该看到什么文案和按钮

它不替代以下文档：

- 架构边界： [ARCHITECTURE.md](/E:/compuser/compuser/ARCHITECTURE.md)
- 支持边界： [PHASE5_VERIFIED_SUPPORT_ENVELOPE.md](/E:/compuser/compuser/PHASE5_VERIFIED_SUPPORT_ENVELOPE.md)
- web-panel 模块边界： [WEB_PANEL_BOUNDARIES.md](/E:/compuser/compuser/apps/web-panel/WEB_PANEL_BOUNDARIES.md)

## 1. 产品目标

普通用户入口不是开放式桌面 agent 聊天框，而是一个高约束的任务办理面板。

这个入口的职责只有三件事：

1. 帮用户判断当前任务是否落在已支持模板内。
2. 在可以执行时明确展示将按哪种模板执行。
3. 在不能执行时明确解释原因，并给出下一步动作。

普通用户入口不承担以下承诺：

- 不承诺支持任意桌面软件
- 不承诺支持任意自然语言任务
- 不承诺支持远程机器或未知 endpoint
- 不承诺开放式长流程桌面自动化

## 2. 首屏结构

首屏应由 4 个区块组成，顺序固定：

1. 顶部标题区
2. 任务输入区
3. 推荐模板区
4. 当前环境状态区

首屏不要展示完整能力库、调试 trace、上下文细节、原始支持矩阵或系统治理细节。

### 2.1 顶部标题区

标题：

`Compuser 任务面板`

副标题：

`只处理当前机器上的已验证任务模板。`

辅助说明：

`如果任务不在支持范围内，我会先说明原因，再给出下一步建议。`

### 2.2 任务输入区

输入框标题：

`描述你要完成的任务`

输入框 placeholder：

`例如：把当前网页内容整理后发到聊天窗口`

输入框下方提示：

`你可以直接描述目标，我会先判断是否属于当前支持模板。`

主按钮：

`开始判断`

次按钮：

`清空输入`

规则：

- 输入后先做任务判定，不直接开跑
- 判定结果必须落到四类状态之一
- 不允许在普通用户入口直接暴露“高级模式”或“全能力列表”

### 2.3 推荐模板区

区块标题：

`推荐任务`

说明文字：

`以下任务已在当前机器和当前环境里验证过。`

每张模板卡固定包含 4 行信息：

- 模板标题
- 一句话说明
- 启动前提摘要
- 主按钮

模板卡主按钮统一用：

`按这个模板开始`

5 张推荐模板卡建议如下：

1. `网页整理后发送到聊天窗口`
   说明：`从当前网页提取内容，整理后发送到已确认的聊天窗口。`

2. `网页内容投递到 Codex`
   说明：`从当前网页提取内容，并发送到本地 Codex 窗口。`

3. `工作区文件填写到网页`
   说明：`读取工作区文件内容，填写到当前浏览器页面。`

4. `比较两个窗口并生成摘要`
   说明：`比较两个已确认窗口的内容，并发送总结结果。`

5. `提取网页内容并轻量改写后发送`
   说明：`提取当前网页内容，整理后投递到已确认目标。`

### 2.4 当前环境状态区

区块标题：

`当前环境`

展示内容只保留普通用户看得懂的 3 项：

- 桌面服务状态
- 当前支持范围
- 最近一次已发布验证结果

建议文案：

- 桌面服务状态：`已连接` / `未连接`
- 当前支持范围：`当前机器上的 5 个已验证任务模板`
- 最近验证结果：`当前已发布验证结果满足 95%+ 支持声明门槛`

不要在这里展示：

- 原始 endpoint 地址细节的长说明
- scorecard 原始 JSON
- capability 列表
- 权限策略表全文

## 3. 提交后的四类状态

用户点击 `开始判断` 或模板卡入口后，界面必须进入以下 4 类状态之一。

### 3.1 状态 A：直接执行

适用条件：

- 已明确命中某个支持模板
- 环境满足执行前提

状态标题：

`可以直接开始`

主文案：

`这项任务已匹配到已验证模板，准备按固定路径执行。`

辅助文案：

`执行路径：{模板名}`

`目标对象：{浏览器 / Codex / 聊天窗口 / 两个窗口}`

主按钮：

`直接开始`

次按钮：

`返回修改`

页面提示：

`开始后会先观察当前环境，再执行任务。`

### 3.2 状态 B：引导改写

适用条件：

- 任务接近某个模板
- 但来源、目标或动作范围仍不够明确

状态标题：

`建议按已支持方式执行`

主文案：

`你的目标接近当前支持模板。为了保证结果可验证，建议先按下面的方式收窄任务。`

建议改写区标题：

`建议执行方式`

建议改写区内容格式：

`从 {来源} 提取内容，经过 {动作}，发送到 {目标}`

主按钮：

`按这个方式执行`

次按钮：

- `我来修改输入`
- `取消`

页面提示：

`普通用户入口只处理已验证模板，不会直接进入开放式桌面控制流程。`

### 3.3 状态 C：环境未就绪

适用条件：

- 模板是对的
- 但当前页面、窗口、权限或桌面服务条件不满足

状态标题：

`当前环境还没准备好`

主文案：

`这项任务属于已支持模板，但当前环境不足以安全执行。`

说明区标题：

`需要先准备`

说明区按 checklist 展示，常见项：

- 打开目标聊天窗口
- 打开本地 Codex 窗口
- 打开可提取文本的网页
- 准备两个可确认窗口
- 连接桌面服务

主按钮：

`我去准备环境`

次按钮：

- `重新检查`
- `返回`

页面提示：

`环境准备好后，可以回到当前页面重新判断。`

### 3.4 状态 D：明确拒绝

适用条件：

- 明显超出当前支持边界
- 不适合通过收窄表达进入现有模板

状态标题：

`当前不支持这项任务`

主文案：

`这项任务不在当前普通用户入口的支持范围内。`

说明区标题：

`当前支持范围`

说明区内容：

- `只支持当前机器上的 5 个已验证模板`
- `只支持浏览器、本地 Codex、已确认聊天窗口`
- `不支持开放式桌面控制、登录流程、未知软件长流程`

主按钮：

`查看支持任务`

次按钮：

- `重新输入`
- `关闭`

页面提示：

`如果你的目标接近网页整理、文件填写、窗口比较或聊天投递，可以改写后再试。`

## 4. 执行中的界面状态

当用户已确认执行后，界面应切到执行态。

执行态只保留 5 个信息块：

- 当前阶段
- 当前模板
- 任务摘要
- 是否需要用户确认
- 紧急中止按钮

执行态标题：

`正在处理`

阶段文案建议：

- `正在观察当前环境`
- `正在确认目标窗口`
- `正在整理任务内容`
- `正在投递结果`
- `正在做最后检查`

紧急中止按钮：

`紧急中止 (Esc)`

说明：

`如果任务明显跑偏，可以立即停止。`

执行态不要展示：

- 原始链路 trace
- 内部能力名
- 低层工具调用详情

## 5. 结果页状态

执行完成后，结果页只分 3 类：

- 成功完成
- 执行失败
- 已中止

### 5.1 成功完成

标题：

`已完成`

主文案：

`任务已按已验证模板执行完成。`

辅助文案：

`如需继续处理，可以基于当前结果发起下一项任务。`

按钮：

- `继续新任务`
- `查看本次结果`

### 5.2 执行失败

标题：

`未完成`

主文案：

`任务没有顺利完成。`

失败说明区必须同时包含两项：

- `失败原因`
- `建议先做什么`

按钮：

- `按建议重试`
- `返回修改任务`

禁止使用的失败风格：

- 只有技术异常文本，没有下一步动作
- 假装成功
- 用内部错误码代替用户可理解说明

### 5.3 已中止

标题：

`已停止`

主文案：

`任务已按你的指令停止。`

辅助文案：

`你可以修改任务后重新开始。`

按钮：

- `重新开始`
- `返回首页`

## 6. 关键按钮文案总表

首屏：

- `开始判断`
- `清空输入`
- `按这个模板开始`

判定态：

- `直接开始`
- `按这个方式执行`
- `我去准备环境`
- `查看支持任务`
- `返回修改`
- `重新输入`

执行态：

- `紧急中止 (Esc)`

- `继续新任务`

- `按建议重试`
- `返回首页`

## 7. 关键用户路径

### 路径 1：直接命中模板

1. 用户输入任务
2. 系统判定为直接执行
3. 用户看到模板和执行预览
4. 用户点击 `直接开始`
5. 系统进入执行态
6. 系统返回成功或失败结果页

### 路径 2：需要改写引导

1. 用户输入模糊任务
2. 系统判定为引导改写
3. 页面展示建议执行方式
4. 用户点击 `按这个方式执行`
5. 系统进入执行态
6. 系统返回成功或失败结果页

### 路径 3：环境未就绪

1. 用户输入支持模板内任务
2. 系统判定为环境未就绪
3. 页面展示需要准备的条件
4. 用户点击 `我去准备环境`
5. 用户准备完成后返回
6. 用户重新判断或重新执行

### 路径 4：明确拒绝

1. 用户输入超出支持面的任务
2. 系统判定为明确拒绝
3. 页面展示当前支持范围
4. 用户选择 `查看支持任务` 或 `重新输入`

## 8. 设计原则

普通用户入口必须一直满足以下原则：

- 先判断，再执行
- 不假装支持未知任务
- 不把环境问题伪装成产品失败
- 不把内部技术结构直接暴露给普通用户
- 每个失败状态都必须告诉用户下一步怎么做

## 9. 后续实现建议

这份交互稿对应的实现顺序建议如下：

1. 先实现四类判定状态的页面分流
2. 再实现每类状态的文案和按钮
3. 再把五个模板的推荐卡与判定规则绑定
4. 最后再打磨视觉层与交互动效

在完成以上步骤之前，不建议先做复杂动画、能力市场式首屏或开放式聊天首页。

## 10. 页面级线框说明

这一节不讨论技术实现，只定义页面上用户会看到什么、先看到什么、能做什么。

### 10.1 首页

首页信息层级固定为 4 层，自上而下不变：

1. 页面标题与一句话边界说明
2. 任务输入区
3. 推荐模板区
4. 当前环境区

页面结构建议如下：

`[标题]`
`Compuser 任务面板`
`只处理当前机器上的已验证任务模板`

`[任务输入区]`
`描述你要完成的任务`
`[ 多行输入框 ]`
`[开始判断] [清空输入]`

`[推荐模板区]`
`5 张推荐模板卡`

`[当前环境区]`
`桌面服务状态 / 当前支持范围 / 最近验证结果`

首页的核心动作只有两个：

- 用户自己输入任务后点击 `开始判断`
- 用户直接点击某个推荐模板卡开始该类任务

首页必须显式压住的信息：

- 不展示完整 capability 列表
- 不展示内部 route、trace、scorecard 原文
- 不展示“高级模式”“全能力模式”入口
- 不展示对普通用户无意义的调试字段

首页视觉重心应放在任务输入区，而不是模板卡列表。模板卡的作用是“帮用户快速对齐支持范围”，不是把首页做成能力市场。

### 10.2 任务判定页

任务提交后，先进入判定页，再决定进入哪一类状态。这个页面不是结果页，而是“是否能做、该怎么做”的解释页。

页面信息顺序固定如下：

1. 判定结论
2. 为什么这样判定
3. 如果能做，将按哪条模板执行
4. 用户下一步按钮

判定页统一骨架如下：

`[状态标题]`
`可以直接开始 / 建议按已支持方式执行 / 当前环境还没准备好 / 当前不支持这项任务`

`[原因说明卡]`
`系统为什么给出这个结论`

`[执行方式或限制说明卡]`
`命中的模板 / 建议改写方式 / 需要准备的环境 / 当前支持范围`

`[底部动作区]`
`主按钮 1 个 + 次按钮 1 到 2 个`

四类状态在同一骨架下变化，不应做成四套完全不同的页面，否则用户会误以为进入了不同系统。

各状态差异如下：

- `直接执行`：主信息是“命中了哪个模板”，主按钮是 `直接开始`
- `引导改写`：主信息是“建议按什么固定表达改写”，主按钮是 `按这个方式执行`
- `环境未就绪`：主信息是“缺什么条件”，主按钮是 `我去准备环境`
- `明确拒绝`：主信息是“为什么超出边界”，主按钮是 `查看支持任务`

判定页必须隐藏的信息：

- 不展示内部打分过程
- 不展示模型推理链路
- 不展示“其实也许能试试”的灰色判断

普通用户页面只能给明确结论，不给模糊中间态。

### 10.3 执行中页

执行中页的目标不是“展示系统很复杂”，而是让用户知道三件事：

1. 任务还在继续
2. 当前做到哪一步了
3. 如果跑偏，用户怎么停

页面结构建议如下：

`[执行状态头部]`
`正在处理`
`当前模板 / 当前阶段`

`[当前阶段卡]`
`正在观察当前环境 / 正在确认目标窗口 / 正在整理任务内容 / 正在投递结果 / 正在做最后检查`

`[任务摘要卡]`
`本次任务会做什么，不超过 3 行`

`[确认提醒区，可选]`
`如果需要用户确认权限或目标，就在这里停住`

`[底部危险动作]`
`紧急中止 (Esc)`

执行中页只保留必要进展，不滚出技术日志流。普通用户不需要看每一次工具调用，只需要知道当前阶段和是否需要自己介入。

执行中页不应出现的内容：

- 原始工具调用回包
- 内部 JSON
- 长日志瀑布流
- 多列监控台式布局

如果确实需要展示更多细节，也只能放在收起区，不应默认展开。

### 10.4 结果页

结果页只回答四个问题：

1. 做完了没有
2. 结果在哪里
3. 如果没做完，卡在哪里
4. 用户下一步该做什么

结果页统一骨架如下：

`[结果标题]`
`已完成 / 未完成 / 已停止`

`[一句话结果说明]`
`任务是否按已验证模板完成`

`[结果摘要区]`
`本次实际产出了什么`

`[下一步动作区]`
`继续新任务 / 查看本次结果 / 按建议重试 / 返回修改任务`

三种结果态重点不同：

- `成功完成`：重心是“结果已产生”，按钮优先 `继续新任务`
- `执行失败`：重心是“失败原因 + 建议先做什么”，不能只给错误
- `已中止`：重心是“任务已停住，可重新开始”，不要制造失败感

结果页可保留一层“查看本次结果”的展开区，但默认先给摘要，不让用户直接面对大段原始输出。

### 10.5 移动端收缩规则

移动端不是重新设计一套产品，只是按优先级收缩。

收缩顺序固定如下：

1. 保留标题、输入区、主按钮
2. 保留当前状态结论
3. 保留一个最关键的解释卡
4. 把推荐模板区折叠为横向滑动卡片或单列卡片
5. 把环境区收进折叠块

移动端必须坚持的规则：

- 主按钮始终可见，不被长说明挤出首屏
- 判定结论必须在首屏内出现
- `紧急中止` 在执行态始终容易点到
- 不做左右双栏

移动端可以牺牲的信息：

- 次级说明文案
- 展开后的细节区
- 次优先模板卡

不能牺牲的信息：

- 当前是四类状态中的哪一类
- 当前下一步动作是什么
- 当前是否需要用户确认

## 11. 页面显隐规则

为了避免普通用户入口失焦，页面元素需要按状态显隐，而不是全部堆在一个面板里。

首页默认显示：

- 标题区
- 输入区
- 推荐模板区
- 当前环境区

判定页默认显示：

- 状态标题
- 原因说明
- 执行方式或限制说明
- 主次按钮

执行中页默认显示：

- 当前阶段
- 任务摘要
- 用户确认区（如有）
- 紧急中止

结果页默认显示：

- 结果标题
- 结果摘要
- 下一步按钮

以下内容只能按需出现，不能常驻：

- 权限确认
- 模板启动前检查明细
- 原始结果全文
- 调试信息

## 12. 首版收口标准

这份前端交互稿如果要进入首版设计或实现，至少应满足以下收口标准：

1. 设计图能清楚画出首页、判定页、执行中页、结果页四种页面
2. 每个页面都有唯一主按钮，不出现多个同级主动作
3. 四类判定状态都能用统一页面骨架表达
4. 普通用户页面不暴露内部技术结构
5. 失败、拒绝、环境未就绪三类状态都能明确告诉用户下一步

如果这 5 条还做不到，就说明交互稿还没收紧，不适合直接进入开发。

## 13. 低保真线框版

这一节用于给产品、设计、前端对齐页面骨架，不定义视觉风格，只定义区块排布与主次关系。

约定：

- `[]` 表示功能区块
- `()` 表示按钮
- 同一行从左到右表示同层级并排
- 从上到下表示阅读顺序

### 13.1 桌面端总框架

首版建议统一使用单主内容区，不做复杂多栏控制台。

```text
+----------------------------------------------------------------------------------+
| [页面标题]                                                [环境状态简写/可选]   |
| Compuser 任务面板                                                             |
| 只处理当前机器上的已验证任务模板                                               |
+----------------------------------------------------------------------------------+
|                                                                                  |
|                                [主内容区]                                        |
|                                                                                  |
| 这里按状态切换：首页 / 判定页 / 执行中页 / 结果页                               |
|                                                                                  |
+----------------------------------------------------------------------------------+
| [辅助区/可折叠]                                                                  |
| 推荐模板 / 环境说明 / 结果详情 / 支持范围                                       |
+----------------------------------------------------------------------------------+
```

桌面端原则：

- 主内容区始终只有一个核心任务
- 辅助区可以折叠，不与主决策抢注意力
- 不出现监控台式三栏并行信息墙

### 13.2 首页低保真线框

```text
+----------------------------------------------------------------------------------+
| [页面标题]                                                                       |
| Compuser 任务面板                                                                |
| 只处理当前机器上的已验证任务模板                                                 |
+----------------------------------------------------------------------------------+
| [任务输入区]                                                                     |
| 描述你要完成的任务                                                               |
| +----------------------------------------------------------------------------+   |
| | 多行输入框                                                                 |   |
| +----------------------------------------------------------------------------+   |
| (开始判断)                                                   (清空输入)          |
+----------------------------------------------------------------------------------+
| [推荐模板区]                                                                     |
| [模板卡 1] [模板卡 2] [模板卡 3]                                                 |
| [模板卡 4] [模板卡 5]                                                            |
+----------------------------------------------------------------------------------+
| [当前环境区]                                                                     |
| 桌面服务状态 | 当前支持范围 | 最近验证结果                                       |
+----------------------------------------------------------------------------------+
```

首页落点：

- 首屏焦点一定在输入区
- 推荐模板只做“帮选”，不做主要入口墙
- 环境区只给状态，不给技术细节

### 13.3 判定页低保真线框

四类判定状态共享同一布局，只替换标题、说明和主按钮。

```text
+----------------------------------------------------------------------------------+
| [返回修改任务]                                                                   |
+----------------------------------------------------------------------------------+
| [状态标题]                                                                       |
| 可以直接开始 / 建议按已支持方式执行 / 当前环境还没准备好 / 当前不支持这项任务   |
+----------------------------------------------------------------------------------+
| [原因说明卡]                                                                     |
| 系统为什么给出这个结论                                                           |
+----------------------------------------------------------------------------------+
| [执行方式或限制说明卡]                                                           |
| 命中的模板 / 建议改写方式 / 缺失条件 / 当前支持范围                             |
+----------------------------------------------------------------------------------+
| [动作区]                                                                         |

+----------------------------------------------------------------------------------+
```

四类状态在这套线框上的落位：

- `直接执行`：第二张卡展示命中模板和执行预览
- `引导改写`：第二张卡展示建议改写句式
- `环境未就绪`：第二张卡展示 checklist
- `明确拒绝`：第二张卡展示边界说明和可回退方向

### 13.4 执行中页低保真线框

```text
+----------------------------------------------------------------------------------+
| [状态头部]                                                                       |
| 正在处理                                                                         |
| 当前模板：XXX                     当前阶段：正在确认目标窗口                     |
+----------------------------------------------------------------------------------+
| [当前阶段卡]                                                                     |
| 当前正在执行哪一步                                                               |
+----------------------------------------------------------------------------------+
| [任务摘要卡]                                                                     |

| 如果需要权限确认或目标确认，就停在这里                                           |
| (允许) (拒绝) / (继续启动) (取消启动)                                            |
+----------------------------------------------------------------------------------+
| [动作区]                                                                         |
|                                                    (紧急中止 Esc)                |
+----------------------------------------------------------------------------------+
```

执行中页落点：

- 先告诉用户“还活着”
- 再告诉用户“现在做到哪”
- 最后给用户“怎么停”

如果没有用户确认事项，就不显示确认区，不拿空盒子占版面。

### 13.5 结果页低保真线框

```text
+----------------------------------------------------------------------------------+
| [结果标题]                                                                       |
| 已完成 / 未完成 / 已停止                                                         |
+----------------------------------------------------------------------------------+

+----------------------------------------------------------------------------------+
| [结果摘要卡]                                                                     |
| 本次实际产出了什么 / 卡在哪里 / 建议先做什么                                     |
+----------------------------------------------------------------------------------+
| [结果详情折叠区，可选]                                                           |
| 查看本次结果                                                                     |
+----------------------------------------------------------------------------------+
| [动作区]                                                                         |
| (继续新任务)                              (查看本次结果) / (按建议重试)          |
+----------------------------------------------------------------------------------+
```

结果页落点：

- 成功时先给“完成感”
- 失败时先给“原因和下一步”
- 中止时先给“可恢复感”

### 13.6 移动端低保真线框

移动端只做纵向堆叠，不保留桌面端并排卡片。

```text
+----------------------------------+
| [页面标题]                        |
+----------------------------------+
| [任务输入区]                      |
| [多行输入框]                      |
| (开始判断)                        |
| (清空输入)                        |
+----------------------------------+
| [当前状态/判定结论]               |
+----------------------------------+
| [最关键说明卡]                    |
+----------------------------------+
| [推荐模板区/折叠]                 |
+----------------------------------+
| [环境区/折叠]                     |
+----------------------------------+
```

移动端落点：

- 先任务，再结论，再解释
- 主按钮整行展示
- 推荐模板和环境信息默认收缩

### 13.7 页面切换关系

```text
[首页]
  -> 开始判断
  -> 点击模板卡

[判定页]
  -> 直接执行
  -> 引导改写
  -> 环境未就绪
  -> 明确拒绝

[执行中页]
  -> 成功完成
  -> 执行失败
  -> 已中止

[结果页]
  -> 继续新任务
  -> 返回首页
  -> 按建议重试
```

页面切换原则：

- 首页只负责发起
- 判定页只负责决定是否能走
- 执行中页只负责展示进展和接用户确认
- 结果页只负责收口和导向下一步

## 14. 出视觉稿前的约束

后续如果进入视觉稿，建议继续保持以下约束：

1. 不把首页画成聊天产品
2. 不把判定页画成复杂分析报告
3. 不把执行中页画成工程监控台
4. 不把结果页画成长日志页
5. 不用“更多能力”“探索模式”之类入口破坏普通用户边界

## 15. 首页与判定页高保真视觉方向

这一节开始定义更接近视觉稿层面的方向，但仍然不进入代码实现或像素级标注。

本轮只收两个页面：

- 首页
- 判定页

原因很直接：这两个页面决定用户会不会相信这个产品，也决定普通用户入口会不会重新滑回“聊天工具”或“工程台”。

### 15.1 整体视觉基调

建议整体基调定义为：

`克制、可信、安静、像任务台而不是聊天框`

视觉气质要求：

- 不做黑客风、终端风、监控台风
- 不做大面积卡片瀑布和能力广场风
- 不做“智能助手人格化”装饰
- 更接近“高质量任务办理台”而不是“聊天陪伴产品”

可以接受的观感关键词：

- 留白明确
- 信息区块分层清楚
- 文案可信，不兴奋，不夸张
- 按钮像操作，不像营销 CTA

### 15.2 首页视觉方向

首页建议采用：

`上部轻标题 + 中部强输入区 + 下部辅助卡区`

视觉重心顺序固定如下：

1. 任务输入框
2. 开始判断按钮
3. 推荐模板
4. 环境状态

页面观感建议：

- 顶部标题区偏轻，不抢输入区
- 输入区是全页最强视觉锚点
- 推荐模板卡要整齐、短、克制，不做促销式卡片
- 环境区退后，作为“让人安心的背景信息”

首页高保真排布建议：

```text
[轻量页头]
Compuser 任务面板
只处理当前机器上的已验证任务模板

[主输入卡 - 页面视觉中心]
描述你要完成的任务
[大输入框]
[开始判断]   [清空输入]

[推荐模板带]
5 张中小卡片，默认一眼看完，不需要滚很长

[环境状态卡]
桌面服务状态 / 当前支持范围 / 最近验证结果
```

### 15.3 首页组件风格建议

标题区建议：

- 标题字号明确，但不过度放大
- 副文案用更轻的灰阶
- 不放 mascot、人格头像、对话气泡

输入区建议：

- 输入卡是页面最大卡片
- 输入框本身要显得“可以放心交代任务”
- 主按钮 `开始判断` 要稳定、明确、像确认动作
- `清空输入` 必须退成次级按钮

推荐模板区建议：

- 每张卡只保留 3 层信息：标题、1 句说明、启动按钮
- 卡片高度尽量统一
- 不在卡片里塞 readiness、score、trace 一类内部信号
- 推荐模板区整体更像“快捷入口”，不是“能力目录”

环境区建议：

- 更像状态说明条，而不是配置面板
- 只讲用户听得懂的话
- 状态错误时也不要变成红色报警台

### 15.4 首页色彩与层级建议

首页不建议用高刺激配色。

建议色彩策略：

- 背景用暖灰或中性浅底
- 主按钮用深色实体按钮
- 成功、警告、拒绝状态色只在少量位置出现
- 默认状态依靠层级和留白，不靠满屏色块

层级建议：

- 一级：输入卡
- 二级：判定主按钮
- 三级：推荐模板卡
- 四级：环境说明

如果视觉稿出现以下情况，说明重心错了：

- 模板卡比输入框更抢眼
- 环境区像主舞台
- 页面看起来像聊天应用启动页

### 15.5 首页动效建议

首页动效只做轻量反馈，不做存在感很强的动画。

建议只保留：

- 首次进入时输入卡轻微上浮或淡入
- 点击 `开始判断` 后按钮进入明确 loading 状态
- 推荐模板 hover 只做轻微阴影或边框变化

不建议：

- 大面积元素飞入
- 对话式打字动画
- 连续脉冲、旋转、炫光

### 15.6 判定页视觉方向

判定页建议采用：

`单结论主屏 + 双说明卡 + 底部动作区`

它的任务不是“展示系统理解了很多”，而是“让用户快速相信这个结论，并知道下一步怎么做”。

视觉重心顺序固定如下：

1. 判定结论
2. 原因说明
3. 执行方式或限制说明
4. 主按钮

判定页高保真排布建议：

```text
[返回入口]
返回修改任务

[主结论卡 - 页面视觉中心]
可以直接开始 / 建议按已支持方式执行 / 当前环境还没准备好 / 当前不支持这项任务

[原因说明卡]
为什么这样判定

[执行方式或限制说明卡]
命中的模板 / 建议改写方式 / 缺失条件 / 支持边界

[底部动作区]
[主按钮]   [次按钮] [次按钮]
```

### 15.7 判定页四种状态的视觉差异

四种状态要共用同一套版式，只通过标题、强调色、按钮文案和第二张卡内容做区分。

建议差异如下：

- `直接执行`
  - 气质：稳定、可确认
  - 强调：命中模板和执行预览
  - 主按钮：最明确
- `引导改写`
  - 气质：纠偏，不否定用户
  - 强调：建议表达方式
  - 主按钮：像“按建议继续”
- `环境未就绪`
  - 气质：缺条件，不是失败
  - 强调：checklist
  - 主按钮：去准备，而不是重试
- `明确拒绝`
  - 气质：边界清楚，但不冒犯
  - 强调：为什么不在支持范围
  - 主按钮：回到支持任务

关键要求：

- `环境未就绪` 不能画得像系统崩了
- `明确拒绝` 不能画得像报错
- `引导改写` 不能画得像用户输错了

### 15.8 判定页组件风格建议

主结论卡建议：

- 是整个页面最大、最稳定的一张卡
- 结论文案要大于说明文案
- 可以配一个很轻的状态图形，但不能喧宾夺主

原因说明卡建议：

- 更像“解释”，不是“分析报告”
- 控制在 2 到 4 行内
- 不出现内部术语

执行方式或限制说明卡建议：

- 对 `直接执行` 和 `引导改写`，重点放在“接下来会怎么做”
- 对 `环境未就绪`，重点放在“先补什么条件”
- 对 `明确拒绝`，重点放在“当前支持什么”

动作区建议：

- 只保留一个真正主按钮
- 其他按钮一律退为次级
- 不要让用户在底部面对 3 个同权重选择

### 15.9 首页与判定页的统一性要求

这两页必须看起来属于同一个产品阶段，而不是两个不同系统。

统一项包括：

- 同一套标题体系
- 同一套按钮层级
- 同一套卡片圆角、边框、留白逻辑
- 同一套状态色使用方式

允许变化的只有：

- 结论强度
- 动作按钮文案
- 第二张说明卡内容

### 15.10 进入视觉稿前的最终检查

首页和判定页的视觉稿出来后，至少要过这 6 个检查：

1. 用户第一眼是否知道“这是任务面板，不是聊天产品”
2. 首页第一视觉焦点是否是输入区，而不是模板卡
3. 判定页第一视觉焦点是否是结论，而不是辅助说明
4. 四类状态是否像“不同结论”，而不是“四种报错”
5. 主按钮是否始终唯一、清晰
6. 页面是否仍然保持克制可信，而不是被设计语言带偏成营销页

如果这 6 条里有两条以上答不上来，视觉稿就还不适合进入实现。

## 16. 现有 web-panel 与目标交互稿的差距清单

这一节的目标不是评价现有实现好坏，而是给后续切页面提供一份可执行的改造清单。

对比基线：

- 目标：本稿定义的普通用户任务面板
- 现状：`apps/web-panel/public/index.html` + `apps/web-panel/public/app.js` 的单页聊天/控制台混合界面

### 16.1 现状一句话判断

当前 `web-panel` 的核心组织方式仍然是：

`左侧会话导航 + 中间聊天线程 + 右侧检查面板`

而目标交互稿要求的是：

`单任务主面板 + 按状态切页 + 最少必要辅助信息`

这不是简单改文案就能对齐的问题，而是信息架构需要从“聊天/控制台视角”切到“任务办理视角”。

### 16.2 一级差距总览

P0 级差距：

- 页面框架错了：现状是三栏工作台，目标是单主面板
- 主隐喻错了：现状是聊天线程，目标是任务办理台
- 状态组织错了：现状把所有状态堆在一个长页面里，目标是首页/判定页/执行中页/结果页切换

P1 级差距：

- 推荐模板位置不对：现在藏在侧栏，目标应进入首页主内容
- 环境信息位置不对：现在像独立运维面板，目标应退到辅助状态区
- 结果与时间线暴露过多：现在默认占主区，目标应按状态显示

P2 级差距：

- 产品命名与文案口径未统一到 `Compuser`
- 视觉气质仍偏 workspace/chat 产品
- 细节面板暴露了过多内部结构

### 16.3 页面结构差距

现状结构：

- 左侧 `app-sidebar` 承担品牌、导航、会话、历史、推荐模板
- 中间 `conversation-panel` 承担欢迎话术、确认区、结果区、时间线、输入区
- 右侧 `inspector-rail` 承担进度、环境、成绩摘要、内部细节

目标结构：

- 一个主内容区
- 按状态切换主页面
- 推荐模板和环境信息作为首页辅助区
- 结果、确认、执行进度只在对应状态页出现

直接结论：

- 当前三栏框架不能原样延续到普通用户入口首版
- 必须先做页面骨架重组，再谈局部视觉优化

### 16.4 首页差距

现状问题：

- 首页没有真正意义上的“首页”
- 输入区被放在页面最底部 `composer-dock`
- 首屏先看到的是聊天说明、会话卡、历史项、时间线容器
- 推荐模板在左侧栏，不是首页主内容
- 环境状态在右侧栏，与输入区分裂

目标要求：

- 输入区必须是首页第一视觉焦点
- 推荐模板必须紧跟输入区出现
- 环境状态应出现在首页底部或辅助区
- 首页不应先展示聊天欢迎语或历史会话

改造动作：

1. 取消“聊天首屏”组织方式
2. 把输入区从底部提升到主内容中心
3. 把推荐模板从侧栏移入首页主内容
4. 把环境状态从右侧 rail 收到首页辅助区

### 16.5 判定页差距

现状问题：

- 当前没有独立判定页
- `render(state)` 会同时刷新模板、时间线、结果、环境、验证等多个区块
- 用户提交任务后，不会先进入“能不能做、该怎么做”的单结论页面
- 判定信息被分散在推荐模板、pending 区、route cards、plan summary 等多个区域

目标要求：

- 提交后必须先进入判定页
- 判定页必须只有一个主结论
- 结论下方只保留两类说明：原因说明、执行方式或限制说明
- 主按钮必须唯一

改造动作：

1. 新增页面级状态路由，而不是继续堆块
2. 把判定相关信息收敛为单独的 `decision view`
3. route cards 不再作为普通用户主视图默认内容
4. pending launch / pending permission 只在进入执行态后按需出现

### 16.6 执行中页差距

现状问题：

- 当前执行态仍然包裹在聊天面板里
- 时间线、结果、确认区同时长期占据主内容
- 执行态没有一个明确的“当前阶段主卡”
- 用户更像在看一个持续滚动的工作台，而不是一个明确执行中的任务

目标要求：

- 执行中页只展示当前阶段、任务摘要、用户确认、紧急中止
- 不默认展示长结果流
- 不让用户在执行过程中同时面对太多旁路信息

改造动作：

1. 将 `timeline` 从主叙事区降级为执行态辅助块
2. 将 `finalOutput` 从执行态主区移出
3. 将确认区与当前阶段区合并到执行态主结构

### 16.7 结果页差距

现状问题：

- 结果区当前常驻在聊天主面板中
- “最后输出”和“完整内容”从页面初始状态就占住固定位置
- 失败、成功、中止没有被组织成独立结果页面

目标要求：

- 结果页只在任务完成后进入
- 默认先看结果摘要，再决定是否展开详情
- 成功、失败、中止使用统一骨架，不做散乱拼接

改造动作：

1. 把 `finalOutputSummary` 和 `finalOutputDetails` 收敛到结果页
2. 将结果页做成单独状态容器
3. 将失败原因与建议动作放到结果摘要核心，而不是散落在其他块

### 16.8 需要删除或降级的现有普通用户首屏元素

以下内容不应继续作为普通用户首屏默认可见内容：

- 左侧历史会话列表
- `Little G` assistant / user 聊天气泡
- “主线程”“今天”“历史”“最近”等会话语境
- 右侧 `成绩摘要`
- 右侧 `内部判断记录`
- 右侧 `操作审批记录`
- 右侧 `工具结果`

这些内容的处理原则：

- 要么删除
- 要么只保留给内部/调试模式
- 要么移入折叠的细节区，不默认展开

### 16.9 可以保留但需要重组的现有能力

以下能力不是问题本身，可以继续复用，但位置与表达必须重组：

- 任务输入 `task-input`
- 文件上传与拖拽 `task-dropzone` / `attachment-list`
- 推荐模板渲染 `renderRecommendedTemplates`
- 权限确认 `renderPendingPermission`
- 模板启动前确认 `renderPendingTemplateLaunch`
- 环境状态 `renderWindowsStatus`
- 结果摘要 `renderFinalOutputSummary`

重组原则：

- 保留数据能力
- 重写页面承载关系
- 不让已有实现结构反过来绑架产品页面结构

### 16.10 命名与文案差距

现状问题：

- 页面标题仍是 `Little G 工作台`
- 存在 `Little G Workspace`、`Ordinary user path`、`主线程` 等 workspace / chat 语义
- 欢迎话术强调“告诉我”“我会先给出建议路线”，仍偏助手对话口吻

目标要求：

- 命名统一为 `Compuser 任务面板`
- 文案统一为任务办理语气
- 去掉人格化助手话术

改造动作：

1. 替换品牌与页头命名
2. 删除聊天欢迎话术
3. 重写按钮和状态文案，使其围绕任务判定与执行

### 16.11 代码结构层面的直接影响

当前前端渲染方式是：

- 一个 `render(state)` 同时刷新几乎所有区块
- 页面没有真正的页面级视图切换
- 各类信息依附在固定 DOM 容器上

要对齐目标交互稿，前端至少需要引入一层新的页面组织：

- `home view`
- `decision view`
- `executing view`
- `result view`

这不一定要求大规模重写状态来源，但要求先重构“状态如何映射到页面”。

直接结论：

- 先改视图组织
- 再改视觉样式
- 最后再压缩内部细节暴露面

### 16.12 建议的切页实施顺序

为了控制风险，建议按下面顺序做：

1. 先移除聊天隐喻和三栏首屏结构
2. 先做首页单主面板
3. 再做判定页分流
4. 再把执行中页从现有时间线/确认区重组出来
5. 最后收口结果页与细节折叠区

不要反过来先修 CSS 或先润色卡片。否则只会把错误的信息架构包得更漂亮。

### 16.13 可直接转成开发任务的清单

任务 A：重建普通用户入口页面骨架

- 去掉左侧历史/会话结构
- 去掉右侧 inspector rail 常驻结构
- 形成单主内容区 + 可折叠辅助区

任务 B：实现页面级状态切换

- 明确首页、判定页、执行中页、结果页四种主视图
- 为四种主视图建立最小渲染入口

任务 C：迁移现有区块到新页面

- 输入区迁到首页
- 推荐模板迁到首页
- pending 确认迁到执行中页
- final output 迁到结果页
- timeline 降级为执行态辅助区

任务 D：删除或隐藏普通用户不该看到的默认块

- 历史会话
- 聊天气泡
- route cards
- 成绩摘要
- 工具结果

任务 E：统一命名与文案

- `Little G` -> `Compuser`
- workspace/chat 语义 -> 任务面板语义

### 16.14 完成差距收口后的验收标准

后续真正切页面后，至少要满足以下标准：

1. 用户首次打开页面时，不会误以为这是聊天产品
2. 用户提交任务后，会先看到单结论判定页
3. 执行中不会默认看到完整结果和大量内部细节
4. 任务完成后，结果页才成为主视图
5. 普通用户默认路径里，看不到内部 route、scorecard、工具结果

如果这 5 条达不到，就说明虽然换了皮，但还没完成从旧 `web-panel` 到目标交互稿的结构切换。
